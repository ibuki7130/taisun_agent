# Requirements: animal-match

> EARS準拠 / C.U.T.E.スコアリング対象 / sdd-full パイプライン生成

## 1. 概要（Executive Summary）

動物占いベースの婚活Webアプリ「アニマルマッチ」は、ユーザーが生年月日とニックネームを入力することで六十干支アルゴリズムにより動物タイプ（12種）を判定し、Claude Haiku APIが生成する結婚性格診断とパートナータイプ提案をワンストップで提供する。フリーミアムモデル（基本診断は無料、マッチング機能は有料1,480円/月）で運営し、MVP（Phase1）を4週間で開発・リリースする。

---

## 2. 背景 & Context

### 現状の課題
- 既存婚活アプリ（IBJ, Pairs, with等）はいずれも動物タイプ分類を採用していない（2026-03-19調査時点）
- 占い×婚活の組み合わせはエンタメ性が高く、SNSシェアによる口コミ獲得が期待できるが、専業サービスが存在しない
- 「楽しい婚活体験」への需要が高まっている（MBTI診断ブーム、性格タイプ自己紹介の定着）

### 制約背景
- 「動物占い®」「個性心理学®」「動物キャラナビ®」は登録商標のため使用禁止
- 六十干支に基づく計算式（数学的事実）は著作権フリーのため使用可能
- 既存プロジェクトのコードベース（Next.js 15、Prisma、TypeScript）を活用

## 8. 機能要件（Functional Requirements）

### REQ-001: ニックネームと生年月日の入力受付

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがニックネーム（1〜20文字）と生年月日（1900-01-01〜診断実行日）を入力して「診断する」ボタンをクリックしたとき、システムは3,000ms以内に動物タイプ計算を完了し、診断結果ページへ遷移しなければならない。
- 根拠/目的: SH-001の主要ユースケース。データ取得の起点。
- 受入テスト(GWT):
  - AT-001a: Given ニックネーム「テスト太郎」、生年月日「1990-01-15」を入力済み When「診断する」をクリック Then 3,000ms以内に `/diagnosis/result/{id}` へ遷移し、動物タイプが「猿」と表示される
  - AT-001b: Given ニックネームが空文字 When「診断する」をクリック Then フォームにバリデーションエラー「ニックネームを入力してください」が表示され、送信されない
  - AT-001c: Given 生年月日が「2050-01-01」（未来日） When「診断する」をクリック Then フォームにバリデーションエラー「生年月日は本日以前の日付を入力してください」が表示される
- 例外・エラー:
  - EH-001a: Claude Haiku APIが5,000ms以内に応答しない場合、システムはフォールバックテキスト（動物タイプ固定文）を表示し、エラーをSentryに記録しなければならない
  - EH-001b: Supabase DBへの書き込みが失敗した場合、システムはユーザーに「診断結果の保存に失敗しました。もう一度お試しください」を表示しなければならない
- 補足:
  - 関連: REQ-002（計算エンジン）、REQ-003（AI生成）
  - トレーサビリティ: SH-001のニーズ「楽しみながら動物タイプを知りたい」

---

### REQ-002: 六十干支計算エンジン

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): 生年月日を受け取ったとき、システムは計算式 `((Math.floor((birthDate.getTime() - new Date(1900,0,1).getTime()) / 86400000) + 8) % 60) + 1` で1〜60の整数（60分類番号）を算出し、所定のマッピングテーブルから12種の動物タイプとグループ（SUN/EARTH/MOON）を返さなければならない。
- 根拠/目的: アプリのコアロジック。計算式は数学的事実であり著作権フリー。
- 受入テスト(GWT):
  - AT-002a: Given 生年月日 `1990-01-15` When 計算を実行 Then 60分類番号=42、動物タイプ=「猿」、グループ=「EARTH」が返る
  - AT-002b: Given 生年月日 `1985-03-20` When 計算を実行 Then 返り値の60分類番号が1〜60の範囲内である
  - AT-002c: Given 同じ生年月日を100回入力 When 計算を実行 Then 毎回同じ動物タイプが返る（決定論的）
  - AT-002d: Given 生年月日 `1900-01-01` When 計算を実行 Then 60分類番号=9（エッジケース確認）
- 例外・エラー:
  - EH-002a: 生年月日が1900-01-01未満の場合、システムは `InvalidDateError` を発生させ、呼び出し元にエラーを返さなければならない
  - EH-002b: 生年月日が診断実行日より未来の場合、システムは `InvalidDateError` を発生させなければならない
- 補足:
  - 関連: REQ-001（入力）、REQ-003（AI生成への入力）
  - トレーサビリティ: SH-001「動物タイプを正確に知りたい」、ASM-001

---

### REQ-003: Claude Haiku APIによる結婚性格診断テキスト生成

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): 動物タイプが確定したとき、システムはClaude Haiku APIを呼び出し、当該動物タイプに対応する結婚性格診断テキスト（性格概要・恋愛傾向・結婚観・理想のパートナー像を各100文字以上300文字以下で含む）を1,000ms以内に生成しなければならない。ただし、同一動物タイプの診断テキストは生成後24時間キャッシュし、API呼び出しを抑制しなければならない。
- 根拠/目的: SH-001への付加価値提供。API呼び出しコストをASM-002の範囲内に抑える。
- 受入テスト(GWT):
  - AT-003a: Given 動物タイプ=「ライオン」 When Claude Haiku APIを呼び出す Then 性格概要・恋愛傾向・結婚観・理想のパートナー像を含む日本語テキストが返る
  - AT-003b: Given 動物タイプ=「ライオン」の診断テキストが24時間以内に生成済み When 同タイプの診断を実行 Then APIを呼び出さずキャッシュからテキストを返し、応答時間が100ms以内である
  - AT-003c: Given Claude Haiku APIが5,000ms以内に応答しない When タイムアウト Then フォールバックテキスト（動物タイプ別の静的テキスト）を返す
- 例外・エラー:
  - EH-003a: Claude Haiku APIがHTTP 429（レート制限）を返した場合、システムはExponential Backoff（最大3回リトライ）を行い、それでも失敗した場合はフォールバックテキストを返しSentryにエラーを記録しなければならない
  - EH-003b: 生成テキストが「動物占い」「個性心理学」「動物キャラナビ」の文字列を含む場合、システムはそれらを「アニマルタイプ」に自動置換しなければならない（商標対応: CON-001）
- 補足:
  - 関連: REQ-002（入力）、CON-004（Haiku限定）
  - トレーサビリティ: SH-001「楽しい診断結果を見たい」、SH-005（Anthropic）

### REQ-004: 相性スコアリングとパートナータイプ提案

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): 診断完了時、システムは60分類番号の円環距離アルゴリズムを用いて全12動物タイプとの相性スコア（0〜100の整数）を算出し、上位3位の推奨パートナータイプと下位1位の注意タイプを診断結果ページに表示しなければならない。
- 根拠/目的: SH-001の「どのような相手を選べばよいか」というニーズに応える。
- 受入テスト(GWT):
  - AT-004a: Given 動物番号=42（猿） When 相性スコアを算出 Then 12タイプ全ての相性スコアが0〜100の整数で返る
  - AT-004b: Given 動物番号=1 When 自身との相性を算出 Then スコア=100が返る（同一タイプは最高相性）
  - AT-004c: Given 動物番号=1 When 動物番号=31（180度対極）との相性を算出 Then スコア=90が返る（補完関係）
  - AT-004d: Given 診断完了 When 結果ページを表示 Then 推奨パートナータイプ3件と注意タイプ1件が表示される
- 例外・エラー:
  - EH-004a: 60分類番号が1〜60の範囲外の場合、システムは `InvalidAnimalNumberError` を発生させ、Sentryに記録しなければならない
- 補足:
  - 関連: REQ-002（スコア入力）、REQ-008（Premiumマッチング）
  - トレーサビリティ: SH-001「理想の相手タイプを知りたい」

### REQ-005: 診断結果のSNSシェア機能

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーが診断結果ページのシェアボタンをクリックしたとき、システムはOGP画像（動物アイコン・動物タイプ名・キャッチコピーを含む1200×630px PNG）を生成し、X（旧Twitter）またはLINEへのシェアリンクを3,000ms以内に提供しなければならない。
- 根拠/目的: SNSシェアによる口コミ拡散でCAC（顧客獲得コスト）を抑制。成功指標: シェア率>=20%。
- 受入テスト(GWT):
  - AT-005a: Given 診断完了（動物タイプ=「ライオン」） When Xシェアボタンをクリック Then `https://twitter.com/intent/tweet?text={エンコード済みテキスト}&url={診断結果URL}` へ遷移する
  - AT-005b: Given 診断完了 When 診断結果ページにアクセス Then `og:image` タグに1200×630pxのOGP画像URLが設定されている
  - AT-005c: Given OGP画像を生成 When 画像URLにアクセス Then PNG形式、1200×630px、ファイルサイズ200KB以下の画像が返る
- 例外・エラー:
  - EH-005a: OGP画像生成に3,000ms以上かかる場合、システムはデフォルト画像（静的ファイル）を使用しなければならない
- 補足:
  - 関連: REQ-001（診断完了が前提）
  - トレーサビリティ: SH-001「友達に診断を教えたい」

### REQ-006: メール認証によるユーザー登録・ログイン

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがメールアドレスとパスワードを入力して「登録」をクリックしたとき、システムはSupabase Auth経由でアカウントを作成し、確認メールを5分以内に送信しなければならない。確認リンクのクリックをもってアカウントが有効化されなければならない。
- 根拠/目的: ユーザー識別と診断履歴の紐付け。Premiumサブスクリプションの前提。
- 受入テスト(GWT):
  - AT-006a: Given 未登録メールアドレスとパスワード（8文字以上、英数字混在） When 登録ボタンをクリック Then 5分以内に確認メールが届き、リンククリック後にダッシュボードへリダイレクトされる
  - AT-006b: Given 登録済みメールアドレス When 登録ボタンをクリック Then 「このメールアドレスは既に登録されています」が表示される
  - AT-006c: Given パスワードが7文字以下 When 登録ボタンをクリック Then 「パスワードは8文字以上で入力してください」が表示され、送信されない
- 例外・エラー:
  - EH-006a: Supabase Auth が503を返した場合、システムは「認証サービスに接続できません。しばらく後にお試しください」を表示しなければならない
- 補足:
  - トレーサビリティ: SH-001「アカウントを作って診断履歴を保存したい」

### REQ-007: Stripeによるサブスクリプション決済

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): 認証済みユーザーが「Premiumプランに登録する」をクリックしたとき、システムはStripe Checkout Sessionを生成し、ユーザーをStripe決済ページへリダイレクトしなければならない。決済成功後、Stripe Webhookを受信した時点で3,000ms以内にユーザーのsubscriptionTierをPREMIUMに更新しなければならない。
- 根拠/目的: 収益化。Stripe Webhookによりデータ整合性を保証する（二重処理防止）。
- 受入テスト(GWT):
  - AT-007a: Given 認証済みFreeユーザー When「Premiumプランに登録する」クリック Then Stripe Checkoutページへリダイレクトされる
  - AT-007b: Given Stripe決済完了 When `checkout.session.completed` Webhookを受信 Then 3,000ms以内にユーザーのsubscriptionTier=PREMIUMに更新され、再ログイン不要でPremium機能が利用可能になる
  - AT-007c: Given 同一のWebhookイベントIDを2回受信 When Webhook処理 Then 2回目は冪等処理され、subscriptionTierが重複更新されない
- 例外・エラー:
  - EH-007a: Stripe Webhookの署名検証に失敗した場合、システムはHTTP 400を返しイベントを無視しSentryに記録しなければならない
  - EH-007b: Stripe Webhookを受信後3,000ms以内にDB更新が完了しない場合、システムはRetryキューに追加し、最大3回再試行しなければならない
- 補足:
  - 関連: REQ-008（Premium機能の前提）
  - トレーサビリティ: SH-002「相性マッチング機能を使いたい」、SH-004（Stripe）

---

### REQ-008: Premiumユーザー向け相性マッチング

- 種別: EARS-状態駆動
- 優先度: MUST
- 要件文(EARS): subscriptionTier=PREMIUMのユーザーがマッチング画面にアクセスしている間、システムは同アプリ内の全Premiumユーザーとの相性スコア上位10件をリストアップし、各ユーザーのニックネーム・動物タイプ・相性スコアを表示しなければならない。
- 根拠/目的: Premiumプランの主要価値。相性の良い相手を探す婚活機能。
- 受入テスト(GWT):
  - AT-008a: Given subscriptionTier=PREMIUM、他のPremiumユーザーが5名存在 When マッチング画面を開く Then 5名全員の相性スコアがリストに表示される（上位順）
  - AT-008b: Given subscriptionTier=FREE When `/matching` にアクセス Then 「Premiumプランへのアップグレードが必要です」画面にリダイレクトされる
  - AT-008c: Given マッチング画面 When ページロード Then 5,000ms以内にリストが表示される（Supabase RLSによるフィルタリング含む）
- 例外・エラー:
  - EH-008a: Premiumユーザーが自分以外に0名の場合、システムは「まだマッチング相手がいません。サービスが広まるのをお待ちください」を表示しなければならない
- 補足:
  - 関連: REQ-007（Premiumサブスク前提）、REQ-004（相性スコア）
  - トレーサビリティ: SH-002「相性スコアで相手を選びたい」

## 9. 非機能要件（Non-Functional Requirements）

### REQ-901: 診断API応答時間（Claude Haiku含む）

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはキャッシュミス時の診断API（`POST /api/diagnosis`）の応答時間をP99で3,000ms以内に維持しなければならない。
- 根拠/目的: ユーザー離脱防止。3秒以上で離脱率が大幅に上昇するため。
- 受入テスト(GWT):
  - AT-901: Given 10並列診断リクエスト（キャッシュなし） When 同時送信 Then P99応答時間が3,000ms以内
- 例外・エラー:
  - EH-901: P99が3,000msを超えた場合、システムはSentryにアラートを送信しなければならない

### REQ-902: システム可用性

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムは30日間の可用性を99.5%（月間ダウンタイム3.6時間以内）に維持しなければならない。
- 根拠/目的: 婚活ユーザーが診断したいタイミングでアクセスできることを保証する。
- 受入テスト(GWT):
  - AT-902: Given 30日間の稼働 When Vercel/Supabaseダッシュボードのアップタイムを確認 Then 可用性が99.5%以上
- 例外・エラー:
  - EH-902: 可用性が99.5%を下回った場合、サービス運営者（SH-003）にメール通知が送信されなければならない

### REQ-903: ページ読み込み時間

- 種別: EARS-普遍
- 優先度: SHOULD
- 要件文(EARS): システムはモバイルデバイス（4G回線想定）における診断結果ページのLCP（Largest Contentful Paint）を2,500ms以内に維持しなければならない。
- 根拠/目的: ASM-006よりスマートフォンユーザーが主体のため、モバイルパフォーマンスを重視。
- 受入テスト(GWT):
  - AT-903: Given Lighthouse（モバイルモード） When 診断結果ページを測定 Then LCPが2,500ms以内
- 例外・エラー:
  - EH-903: LCPが2,500msを超える場合、Next.js Image最適化の設定を見直しなければならない

## 10. セキュリティ要件

### REQ-SEC-001: Row Level Security（RLS）によるデータアクセス制御

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはSupabaseの全テーブル（users, diagnosis_results, compatibility_results, matches, subscriptions）にRow Level Securityを有効化し、認証済みユーザーが自分自身のデータのみ読み書きできるようにしなければならない。
- 根拠/目的: ユーザーデータの不正アクセス防止（CON-002）。
- 受入テスト(GWT):
  - AT-SEC-001a: Given ユーザーA（id=1）でログイン When `GET /api/diagnosis/{user_b_diagnosis_id}` を実行 Then HTTP 403が返り、ユーザーBの診断結果が取得できない
  - AT-SEC-001b: Given Supabase Studio When usersテーブルのRLSポリシーを確認 Then `auth.uid() = id` のSELECTポリシーが存在する
- 例外・エラー:
  - EH-SEC-001: RLSポリシーが無効化されているテーブルが検出された場合、CI/CDパイプラインはデプロイを停止しなければならない
- 補足:
  - トレーサビリティ: CON-002（個人情報保護法対応）

---

### REQ-SEC-002: Stripe Webhook署名検証

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): Stripe Webhookを受信したとき、システムはStripe提供の署名検証（`stripe.webhooks.constructEvent`）を実行し、署名が不正な場合はHTTP 400を返し処理を中断しなければならない。
- 根拠/目的: 悪意ある第三者によるWebhook偽装でのサブスクリプション不正昇格を防止する。
- 受入テスト(GWT):
  - AT-SEC-002a: Given 正しいStripe署名ヘッダー When Webhook受信 Then HTTP 200が返り処理が実行される
  - AT-SEC-002b: Given 不正な署名ヘッダー When Webhook受信 Then HTTP 400が返り処理が実行されず、Sentryにログが記録される
- 例外・エラー:
  - EH-SEC-002: `STRIPE_WEBHOOK_SECRET` 環境変数が未設定の場合、アプリケーションは起動時に `ConfigurationError` を発生させ起動を停止しなければならない
- 補足:
  - 関連: REQ-007（Webhook処理）

---

### REQ-SEC-003: 環境変数による秘密情報管理

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはAPIキー（ANTHROPIC_API_KEY, STRIPE_SECRET_KEY, SUPABASE_SERVICE_ROLE_KEY）を環境変数からのみ取得しなければならない。これらの値はソースコードに直接記述してはならず、`.env.local` はGitリポジトリにコミットしてはならない。
- 根拠/目的: 秘密情報の漏洩防止（CON-002）。
- 受入テスト(GWT):
  - AT-SEC-003a: Given GitHubリポジトリ When 全コミット履歴を `sk-ant` / `sk_live` / `service_role` で検索 Then マッチするコミットが0件
  - AT-SEC-003b: Given `.gitignore` When 内容を確認 Then `.env.local` が除外設定されている
- 例外・エラー:
  - EH-SEC-003: アプリ起動時にいずれかの必須環境変数が未設定の場合、システムは起動を中断し `Missing required env: {KEY_NAME}` をstderrに出力しなければならない
- 補足:
  - トレーサビリティ: CON-002（個人情報保護法）、SH-005（Anthropic APIキー）

---

### REQ-SEC-004: 個人情報の最小収集と保護

- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはユーザーの本名を収集してはならない。ニックネーム（1〜20文字）のみを表示名として使用しなければならない。生年月日は暗号化してSupabaseに保存しなければならない（Supabase encryption at rest有効化）。
- 根拠/目的: 個人情報保護法第16条（利用目的の制限）・第17条（情報の正確性確保）への準拠。婚活データという要配慮情報の性質上、最小化が重要。
- 受入テスト(GWT):
  - AT-SEC-004a: Given ユーザー登録フォーム When フォームフィールドを確認 Then 「氏名」フィールドが存在しない
  - AT-SEC-004b: Given Supabaseプロジェクト設定 When Encryptionの設定を確認 Then Encryption at restが有効（有効化済みの場合は変更不可のため設定確認で代替）
- 例外・エラー:
  - EH-SEC-004: 本名に該当するフィールド名（`fullName`, `lastName`, `firstName`）がDBスキーマに追加されたPRがマージされた場合、コードレビュアーはマージを拒否しなければならない
- 補足:
  - トレーサビリティ: CON-002（個人情報保護法）、SH-001

---

## 11. ログ/監視/運用要件

### REQ-LOG-001: エラーログの集中管理

- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): アプリケーション内で未処理の例外が発生したとき、システムはSentry（DSNはSENTRY_DSN環境変数から取得）にエラーを送信しなければならない。送信データにはユーザーID（匿名化済み）、エラースタックトレース、発生日時を含み、平文の生年月日・メールアドレスを含んではならない。
- 根拠/目的: インシデント対応の迅速化。個人情報のログへの混入を防ぐ。
- 受入テスト(GWT):
  - AT-LOG-001a: Given 意図的なRuntimeError When 発生 Then Sentryダッシュボードに3,000ms以内に記録される
  - AT-LOG-001b: Given Sentryに送信されるエラーログ When 内容を確認 Then 生年月日・メールアドレスの平文が含まれない
- 例外・エラー:
  - EH-LOG-001: Sentryへの送信に失敗した場合、システムはコンソールにエラーを出力し、ユーザー向けの操作は継続しなければならない
- 補足:
  - 関連: REQ-SEC-003（個人情報保護）
